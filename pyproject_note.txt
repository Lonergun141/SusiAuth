# 2. Critical current issues to inspect and fix

Explicitly investigate and resolve the following existing risks.

## Refresh-token defects

Verify whether login currently returns the stored refresh-token hash instead of the raw token.

The correct behavior must be:

1. Generate a cryptographically secure raw refresh token.
2. Hash the token before database storage.
3. Return the raw token exactly once.
4. Never expose the stored hash to the client.
5. Never log either the raw token or stored hash.

Fix refresh rotation so that it does not:

* Create an unnecessary new session
* Create an unused refresh token
* Create duplicate access tokens
* Return a token unrelated to the rotated session
* Allow two concurrent rotations to succeed
* Lose the token-family relationship

Refresh rotation must be:

* Atomic
* Transactional
* Protected with database row locking
* Safe against race conditions
* Safe across multiple service instances

Use `transaction.atomic()` and `select_for_update()` or an equivalent concurrency-safe design.

## Session defects

Ensure:

* Login creates exactly one session.
* Refresh continues the same session.
* Single-session logout revokes only the intended session.
* Logout-all revokes every active session and refresh token.
* Revoked sessions cannot refresh.
* Inactive sessions cannot issue new access tokens.
* Session expiration is enforced.
* Token-family reuse revokes the affected family or session.
* Administrative session revocation works correctly.

## Rate-limiting defects

Do not use Django local-memory cache for production rate limiting.

Process-local rate limiting is invalid in a multi-container environment because each application instance maintains separate counters.

Use Redis-backed distributed rate limiting.

## Email defects

Remove unmanaged thread-based email sending.

Do not use:

* `ThreadPoolExecutor`
* Fire-and-forget Python threads
* Request-process background work
* Direct network calls inside active database transactions

Use:

* Celery
* Redis
* Resend
* Idempotency keys
* Explicit retry policies
* Structured delivery records
* Verified webhooks

## Docker defects

Do not run:

```bash
python manage.py runserver
```

inside a production container.

Do not run migrations automatically from every web replica.

Use:

* Gunicorn
* Appropriate WSGI or ASGI workers
* A dedicated migration command or one-off migration container
* Non-root runtime user
* Multi-stage Docker image
* Health checks
* Graceful shutdown

## Production-setting defects

Production must not accept:

* Default Django secret keys
* Wildcard allowed hosts
* Missing Resend API key
* Missing Resend webhook secret
* Missing JWT signing keys
* Debug mode
* Unverified frontend URLs
* Hardcoded local URLs
* Insecure cookie settings
* Untrusted proxy headers

The service must fail fast when required production configuration is missing.

# 8. Refresh tokens and sessions

## Refresh-token model

Store:

* UUID
* User
* Session
* OAuth client
* Token hash
* Token family UUID
* Parent token
* Replacement token
* Created timestamp
* Last-used timestamp
* Absolute expiration
* Inactivity expiration
* Revoked timestamp
* Revocation reason
* IP metadata
* User-agent metadata

Only the cryptographic hash may be stored.

## Rotation flow

The refresh flow must:

1. Hash the supplied raw token.
2. Lock the matching database row.
3. Verify the token exists.
4. Verify it is not expired.
5. Verify it is not revoked.
6. Verify its session is active.
7. Verify its client is active.
8. Revoke the current token.
9. Create one replacement token in the same family and session.
10. Create one new access token.
11. Commit atomically.
12. Return the new raw refresh token once.

## Reuse detection

When an already rotated or revoked refresh token is reused:

* Record a security event.
* Revoke its token family.
* Revoke the related session.
* Optionally revoke all user sessions according to configured policy.
* Notify the account owner through Resend.
* Do not reveal sensitive internal details to the caller.

## Concurrency

Add an integration test that sends two simultaneous refresh requests using the same token.

Exactly one request may succeed.

The other must fail and trigger the configured reuse-detection response.

## Celery delivery

All production email sending must happen through Celery.

1. Complete database changes inside a transaction.
2. Create the outbound email record.
3. Use `transaction.on_commit()` to enqueue the task.
4. Pass only the outbound email UUID to Celery.
5. Load the record inside the worker.
6. Render the template.
7. Send through Resend.
8. Store the Resend email ID.
9. Update delivery status.
10. Record sanitized errors.
11. Retry only retryable failures.

Do not pass:

* Django model instances
* Raw tokens

# 25. Acceptance criteria

The task is complete only when all of the following are verified.

## Application

* The project starts from a clean checkout.
* Documented Docker commands work.
* PostgreSQL starts successfully.
* Redis starts successfully.
* Web service starts successfully.
* Celery worker starts successfully.
* Migrations run through a deliberate deployment step.
* Production uses Gunicorn or an appropriate production server.
* No container uses Django `runserver` in production.

## Authentication

* Registration works.
* Email verification works.
* Login works.
* Password reset works.
* Password change works.
* MFA works.
* Session listing works.
* Single-session logout works.
* Logout-all works.
* Suspended accounts are blocked.

## Tokens

* Raw refresh tokens are returned only once.
* Only refresh-token hashes are stored.
* Rotation creates exactly one replacement token.
* Rotation maintains the same session.
* Concurrent refresh attempts cannot both succeed.
* Reuse detection revokes the correct session or family.
* JWT claims are validated correctly.
* JWKS works.
* Signing-key rotation is documented and tested.

## OAuth/OIDC

* Authorization Code with PKCE works.
* Client Credentials works.
* Discovery metadata works.
* JWKS is discoverable.
* Registered redirect URI enforcement works.
* Client scopes are enforced.
* Client secrets are hashed.

## Resend

* Production uses Resend.
* Resend API keys are never committed.
* Production requires a verified sending domain.
* Authentication emails have HTML and plain-text versions.
* Email tasks run through Celery.
* Email tasks are dispatched after transaction commit.
* Idempotency keys prevent duplicate sends.
* Resend provider email IDs are persisted.
* Webhook signatures are verified using the raw body.
* Duplicate webhook events are ignored safely.
* Out-of-order events are handled.
* Bounce events are processed.
* Complaint events are processed.
* Failed and suppressed events are processed.
* No OTP, reset token, or API key appears in logs.

## Security

* Redis-backed rate limiting works across instances.
* Account enumeration protections pass.
* Production configuration fails closed.
* Django deployment checks pass.
* Security headers are configured.
* Private signing keys are not in Git.
* Secrets are not present in Docker images.
* Audit events are recorded.

## Quality

* Tests pass.
* Coverage threshold passes.
* CI passes.
* Docker image builds.
* Documentation matches the implementation.
* Remaining limitations are documented honestly.

Begin by auditing the repository. Produce the current-state report and implementation plan before changing architecture. Then implement the improvements in risk-priority order and verify every acceptance criterion with tests or documented deployment checks.
